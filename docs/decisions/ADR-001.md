# ADR-001: Why we chose Next.js

**Status:** Accepted

**Date:** 2026-07-20

## Context

We needed a frontend framework for AI Hunter's web application that supports:
- Server-side rendering for performance and SEO
- API routes co-located with the frontend
- TypeScript out of the box
- Strong ecosystem and community support
- Path to production-grade deployment

## Decision

We will use Next.js with the App Router for `apps/web`.

## Rationale

1. **Full-stack React** — Next.js allows us to build the frontend and API layer in a single framework, reducing context-switching.
2. **App Router** — The modern React paradigm with server components, streaming, and nested layouts.
3. **Ecosystem** — Vercel, Turborepo, and the React ecosystem all align around Next.js.
4. **Performance** — SSR, static generation, and edge runtime give us flexibility for every page type.
5. **Team productivity** — One language (TypeScript), one framework, fewer dependencies to manage.

## Consequences

- The API layer will be built as Next.js API routes initially, with the option to extract into `apps/api` if needed.
- We commit to the App Router pattern and React Server Components.
- We accept Vercel as the primary deployment target, with flexibility for self-hosting.

## Alternatives Considered

- **Remix** — Strong contender, but smaller ecosystem and fewer freelancer-specific examples.
- **Plain Vite + Express** — More flexible but requires more decisions; slower to iterate.
- **SvelteKit** — Less ecosystem support for AI/ML integrations.
